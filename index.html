What is 0/1 Knapsack? — start here if the problem is new to you ▸
The problem

You have n items, each with a weight and a value. You also have a knapsack with a fixed capacity W (the maximum total weight it can carry). Your goal is to pack items into the knapsack to maximize total value — but every item is all-or-nothing: you either take it or leave it. You can't take half an item. That's the "0/1" — each item's indicator is either 0 (skipped) or 1 (taken).

Two versions of the problem

This tool focuses on the decision version, which asks a yes/no question rather than a maximization question:

  • Optimization version: "What is the maximum total value achievable without exceeding W?"
  • Decision version: "Does there exist a subset with total weight ≤ W and total value ≥ V?" (where V is a target value you provide)

The two versions are computationally equivalent — if you can solve one quickly, you can solve the other quickly. But the decision version is what NP-completeness is defined for, which is why the proof on this page targets the decision version specifically.

Where it shows up in practice
  • Project selection: items = projects, weight = budget required, value = expected return, W = total budget
  • Cargo loading: items = packages, weight = actual weight, value = shipping revenue, W = truck capacity
  • Resource allocation: items = tasks, weight = hours needed, value = importance, W = hours available
  • Cutting stock, portfolio selection, and many scheduling problems reduce to Knapsack or its variants
Why "0/1" matters — the fractional cousin

There's a related problem called Fractional Knapsack, where you can take pieces of items (like cutting a bar of gold in half). Fractional Knapsack is easy — a simple greedy algorithm (take items in order of value-per-weight ratio) always finds the optimum. But the moment you forbid fractions, greedy breaks, and the problem jumps to NP-complete. This tool's reduction proof shows why.

Reading this tool

Page 1 (this page) proves 0/1 Knapsack is NP-complete: first by building a polynomial-time verifier for it (showing it's in NP), then by transforming Subset Sum — another NP-complete problem — into it (showing it's at least as hard). Page 2 then shows three concrete algorithms that solve it: brute-force (always correct, exponential), dynamic programming (always correct, pseudo-polynomial), and greedy (fast but not always correct).

Why the O(nW) DP doesn't contradict NP-completeness

The DP algorithm fills a table with n × W cells, so it runs in O(nW) time. That looks polynomial — so why is Knapsack still NP-complete?

The key is that W is a number, encoded in the input using about log₂(W) bits. A truly polynomial-time algorithm must be polynomial in the bit-length of the input, not in the numeric value of W. O(nW) grows exponentially with the number of bits used to encode W:

Capacity W Bits to encode W DP operations (n=10)
10 ~4 bits 100
100 ~7 bits 1,000
10,000 ~14 bits 100,000
1,000,000,000 ~30 bits 10,000,000,000

Notice: input size grows by a handful of bits while operations explode by 10× per row. This is called pseudo-polynomial time — fast when numbers are small, impractical when they're large. Knapsack is weakly NP-complete because its hardness depends on large numeric inputs, so the DP doesn't contradict the proof.

!
Key concept — "polynomial" is measured in bits, not values
A natural mistake: "DP is O(nW), that's polynomial, so Knapsack is easy." Wrong. In complexity theory, "polynomial time" means polynomial in the number of bits it takes to write the input down. Since W is a single number, writing it takes only log₂(W) bits — doubling W adds just one bit to the input, but doubles the number of DP operations. That's the pseudo-polynomial trap.

Algorithm Verdict — When to Use Which?

The live results above tell you what each algorithm did on this input. The reference below tells you which algorithm to reach for in general.
Algorithm Time complexity Always correct? Best when…
Brute-Force O(n · 2ⁿ) YES (exact) n is very small (≤ 20)
Dynamic Programming O(n · W) YES (exact) W is small relative to 2ⁿ
Greedy O(n log n) NO (counterexample) Speed matters and approximation is acceptable
Brute-Force
Tries every possible subset, so it always finds the right answer. The 2ⁿ growth rate makes it impractical for large inputs (at n = 30 you're already at a billion subsets), but for small problems it's the safest choice because there is nothing it can miss.
Dynamic Programming
Always finds the optimal answer by reusing solutions to smaller sub-problems. Runs in O(n · W), which is fast when W is small but explodes when W is large. The bound is "pseudo-polynomial" — polynomial in the numeric value of W, not in the bit-length of W — which is why DP doesn't contradict 0/1 Knapsack being NP-complete.
Greedy (by value-to-weight ratio)
Sorts items by v/w and takes them greedily until full. Very fast, but not always correct — the "Greedy fails" preset shows a 3-item case where greedy gets only 73% of the optimal value. Plain greedy has no provable approximation ratio for 0/1 Knapsack at all; ratios can be arbitrarily bad. The standard fix — taking the better of greedy's answer and the single most valuable item that fits — gives a provable 1/2-approximation.

Educational Content

OverviewWhat is 0/1 Knapsack?

The 0/1 Knapsack problem asks: given a set of items, each with a weight and a value, and a knapsack with a fixed weight capacity W, which items should you pack to get the highest total value without exceeding W? The "0/1" means each item is either fully included or fully excluded — you can't take half of an item.

Decision vs. Optimization

Two versions of the same problem:

  • Optimization: "What is the maximum total value achievable?" — returns a number.
  • Decision: "Is there a selection with total value ≥ V, using weight ≤ W?" — returns YES or NO.

The decision version is what NP-completeness theory talks about, because NP is defined in terms of yes/no problems. A fast solver for either version gives a fast solver for the other, so they're equally hard.

When you'd use it in real life
  • Project portfolio selection: each project has a cost (weight) and expected return (value); budget is the capacity.
  • Cargo loading: fit the most valuable cargo into a truck or ship with a weight limit.
  • Resource allocation: assign limited CPU/memory/time to tasks with different payoffs.
  • Cryptography (historical): the Merkle-Hellman knapsack cryptosystem was based on this problem — later broken, but the idea lives on.
0/1 vs. Fractional Knapsack

The fractional variant — where you can take any fraction of an item — is actually easy: greedy-by-ratio always finds the optimum in O(n log n) time. That's why greedy fails on 0/1 Knapsack: the "just take a slice" escape hatch is gone, and the discrete all-or-nothing choices are what make the problem NP-complete.

Step 2 — Dynamic Programming builds a table O(n · W)

DP defines dp[i][w] = best value using the first i items within weight w. Each cell is either skip item i (copy from dp[i-1][w]) or take item i (add its value to dp[i-1][w-wᵢ], if it fits). The final cell at the bottom-right is the answer.

DP answer: 220. Same as brute-force (as expected — both are exact). Trace-back from dp[3][50] = 220 recovers items 2 and 3.

Step 3 — Greedy picks by value/weight ratio O(n log n)

Greedy sorts items by their value-per-weight ratio and takes them in order as long as they fit.

  • item 1: 60/10 = 6.0 per unit
  • item 2: 100/20 = 5.0 per unit
  • item 3: 120/30 = 4.0 per unit

Sorted order: item 1, item 2, item 3.

  • Take item 1 → weight 10, value 60. Capacity remaining: 40.
  • Take item 2 → weight 30, value 160. Capacity remaining: 20.
  • Try item 3: needs 30, only 20 left → skip.

Greedy answer: 160, taking items 1 and 2.

Key Concepts
Greedy-choice property fails
A problem has the greedy-choice property if you can reach an optimal solution by making a locally optimal choice at each step. 0/1 Knapsack does not — the worked example above shows why. Fractional Knapsack does, which is why greedy works there.
Optimal substructure — why DP works
The best answer for n items and capacity W can be built from best answers for fewer items and smaller capacities: dp[i][w] = max(dp[i-1][w], dp[i-1][w-wᵢ] + vᵢ). Each subproblem has a single optimal value, and the whole solution composes from them. This is what makes dynamic programming possible.
Pseudo-polynomial, not polynomial
DP runs in O(n · W), which looks polynomial — but W is encoded in log₂(W) bits, so this is exponential in the input's bit-length. This is why Knapsack is NP-complete even though DP exists. The full treatment is on the NP page.
Common mistake: "brute-force is always bad"
Not always. For small n, the 2ⁿ factor is tiny — at n = 10 it's 1024, which your computer finishes in microseconds. The growth chart above shows brute-force only diverges from DP and greedy past around n = 20. Know when to use each.
